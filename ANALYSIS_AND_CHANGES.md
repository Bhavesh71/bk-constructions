## 3. FINANCIAL DATA MODEL

### 3.1 Core Financial Rule

```
DailyRecord.grandTotal = totalMaterial + totalOther + totalLabour

Where:
  totalMaterial = sum of MaterialEntry.total for this record
  totalOther    = sum of OtherExpense.amount (includes Advance category)
  totalLabour   = sum of PAID LabourEntry.cost (starts 0, updated on payment)
```

### 3.2 Labour Payment Flow

```
Step 1: Mark Attendance
  → LabourEntry created (isPaid=false, cost=dailyRate)
  → DailyRecord totals UNCHANGED (totalLabour stays 0)
  → Workers are WORKING but not yet PAID

Step 2: (Optional) Give Advance
  → LabourAdvance record created
  → OtherExpense(category='Advance') created on the SAME day's DailyRecord
  → DailyRecord.totalOther INCREASES immediately
  → DailyRecord.grandTotal INCREASES immediately
  → Advance appears in daily spending RIGHT AWAY

Step 3: Weekly Settlement (Pay button)
  → WeeklySalary record created
  → ALL unpaid LabourEntries for the worker marked isPaid=true
  → For each affected DailyRecord:
      totalLabour = SUM(cost) of all paid LabourEntries for that record
      grandTotal  = totalMaterial + totalOther + totalLabour
  → Selected advances marked isSettled=true (linked to WeeklySalary)
  → netPaid = totalWage - advanceDeducted (advance deductions apply to CASH only)
```

### 3.3 Critical Financial Distinction

| Item | Hits grandTotal? | When? |
|------|-----------------|-------|
| Material expenses | YES | Immediately when saved |
| Other expenses | YES | Immediately when saved |
| Labour advances | YES | Immediately when advance is given |
| Unpaid labour wages | NO | Not until salary is paid |
| Paid labour wages | YES | When Pay Week is recorded |

### 3.4 Advance Deduction Logic

Advance deductions affect **cash in hand** (WeeklySalary.netPaid), NOT the
DailyRecord's labour cost.

Example:
- Worker worked 5 days × ₹500 = ₹2,500 totalWage
- Worker received ₹500 advance (already in DailyRecord.totalOther)
- netPaid = ₹2,500 − ₹500 = ₹2,000 (cash given to worker)
- DailyRecord.totalLabour = ₹2,500 (true cost to site)
- The ₹500 advance was already counted on the day it was given

This ensures: Site budget = TRUE cost of labour, not reduced by advance accounting.

## 4. BUGS FOUND & FIXED

### BUG-1 (CRITICAL): Advance OtherExpense lookup uses wrong record

**File**: `src/actions/labour.ts`
**Functions**: `editLabourAdvance`, `deleteLabourAdvance`

**Problem**:
```typescript
// OLD CODE — WRONG
const linked = await tx.otherExpense.findFirst({
  where: {
    dailyRecordId: advance.dailyRecord.id,
    category: 'Advance',
  },
  orderBy: { createdAt: 'desc' },  // ← finds MOST RECENT advance, not THIS advance
})
```

When multiple workers receive advances on the same site/date, editing Worker A's
advance would find Worker B's OtherExpense (the most recently created one) and
corrupt **Worker B's** expense record.

**Impact**: Data corruption — wrong OtherExpense amounts get updated/deleted.
Both `editLabourAdvance` AND `deleteLabourAdvance` had this bug.

**Fix**: Embed the LabourAdvance ID in the OtherExpense description:
```
[ADV:cld_abc123] Advance to Ramu — Medical
```
Then look up by this marker:
```typescript
// NEW CODE — CORRECT
const precise = await tx.otherExpense.findFirst({
  where: {
    dailyRecordId,
    category: 'Advance',
    description: { contains: `[ADV:${advanceId}]` },
  },
})
```

A legacy fallback also handles records created before this fix.

### BUG-2 (SIGNIFICANT): Week boundaries use local time instead of UTC

**File**: `src/actions/labour.ts`
**Function**: `getWeekBounds()`

**Problem**:
```typescript
// OLD CODE — WRONG
const start = new Date(refDate)
start.setDate(refDate.getDate() + diff)
start.setHours(0, 0, 0, 0)  // ← local midnight, not UTC midnight
```

Records are stored at UTC midnight (via `Date.UTC()`), so all date boundary
queries must also use UTC boundaries. On IST (UTC+5:30) servers, local midnight
= 18:30 UTC of the previous day — causing queries like "this week's attendance"
to miss records or include wrong ones.

**Fix**:
```typescript
// NEW CODE — CORRECT
function getWeekBoundsUTC(refDate = new Date()) {
  const dayUTC    = refDate.getUTCDay()
  const diffToMon = dayUTC === 0 ? -6 : 1 - dayUTC
  const weekStart = new Date(Date.UTC(
    refDate.getUTCFullYear(), refDate.getUTCMonth(),
    refDate.getUTCDate() + diffToMon,
  ))
  // ...
}
```

### BUG-3 (MINOR): WeeklySalary weekStart/weekEnd display used local date components

**File**: `src/actions/labour.ts`
**Function**: `getUnpaidSummary()`

**Problem**: `toLocalDateStr` used `dt.getFullYear()`, `dt.getMonth()`, `dt.getDate()`
(local time), but weekStart was computed from `setHours(0,0,0,0)` (local midnight),
creating inconsistency.

**Fix**: Use UTC getters consistently:
```typescript
function toDateStr(dt: Date): string {
  return `${dt.getUTCFullYear()}-${String(dt.getUTCMonth() + 1).padStart(2,'0')}-${String(dt.getUTCDate()).padStart(2,'0')}`
}
```

---

### BUG-4: Monthly trend chart used local month instead of UTC month

**File**: `src/actions/daily-records.ts`
**Function**: `getDashboardData()`

**Problem**:
```typescript
const key = `${monthNames[d.getMonth()]} ...`  // ← local month
```

For UTC-stored dates, the UTC month getter must be used:
```typescript
const key = `${monthNames[dt.getUTCMonth()]} ...`  // ← UTC month
```

## 5. ARCHITECTURAL DECISIONS

### 5.1 No schema changes
The existing Prisma schema is preserved exactly. All fixes are implemented
in application logic only.

### 5.2 OtherExpense ↔ LabourAdvance linkage
Since the schema has no FK between these tables, the "ID embedding in description"
approach is used. This is a non-destructive convention — existing records without
the marker fall back to the legacy matching strategy.

### 5.3 totalLabour = GROSS wages
DailyRecord.totalLabour reflects the GROSS labour cost (rate × days), not the
net-of-advance cash paid. This is correct because:
- Site budget analysis should show TRUE cost, not accounting adjustments
- The advance was already counted on the day it was given
- netPaid is recorded on WeeklySalary for cash-flow tracking

### 5.4 Advance always linked to a site+date
The GiveAdvance modal REQUIRES a site and date, ensuring every advance is
immediately reflected in a DailyRecord and visible in the budget.

## 7. DATA MIGRATION NOTE

Existing LabourAdvance records in production do NOT have the `[ADV:<id>]` marker
in their linked OtherExpense descriptions. The legacy fallback in
`findLinkedOtherExpense` handles these gracefully by matching on amount + labour name.

New advances created after deploying v7 will use the precise ID-encoded marker,
eliminating any ambiguity going forward.
